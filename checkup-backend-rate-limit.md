# Backend 무차별 대입 공격(Brute Force) 방어 — IP Rate Limit

> 자체 서버(Spring Boot · 1대 · 인메모리) 기준
> 작성일: 2026-06-19

---

## 1. 개요 & 목적

- 동일 클라이언트(IP)가 짧은 시간에 인증/조회 요청을 반복하는 **무차별 대입(Brute Force) · 자동화 공격**을 1차로 차단한다.
- 모든 요청이 비즈니스 로직 이전에 이 단계를 통과한다.
- PoW 챌린지(Altcha)와 **계층적으로 함께 동작**한다. (Rate Limit = 양·속도 제한 / PoW = 자동화 비용 상승)

---

## 2. 처리 흐름

```
요청 수신 (클라이언트 IP 추출)
        │
        ▼
[ IP Rate Limit 체크 — 차단 중인가? ]
        │
   YES ─┴─▶ HTTP 429 응답 (Too Many Requests)
        │
    NO  ▼
   비즈니스 로직 처리
```

| 단계 | 동작 |
|---|---|
| 요청 수신 | 클라이언트 IP 추출 |
| Rate Limit 체크 | 해당 IP가 현재 차단 상태인지, 카운터 임계 초과 여부 판정 |
| 차단 중(YES) | **HTTP 429** 응답 (이하 응답 사양 참고) |
| 정상(NO) | 비즈니스 로직으로 진행 |

---

## 3. 점진적 차단 정책

반복 위반 시 차단 시간을 단계적으로 늘려 정상 사용자 영향은 최소화하고 공격자 비용은 누적시킨다.

| 단계 | 조건 | 차단 시간 |
|---|---|---|
| 1단계 | 10분 내 10회 초과 | **5분 차단** |
| 2단계 | 차단 해제 후 재초과 | **15분 차단** |
| 3단계 이상 | 반복 재초과 | **30분 차단 (상한)** |

- **최대 차단 시간: 30분** (상한)
- 임계치(10분/10회), 단계별 차단 시간은 서버 설정으로 조정 가능

---

## 4. 구현 방식 (인메모리)

- **인메모리 `ConcurrentHashMap`** 으로 IP별 카운터·차단 상태를 빠르게 처리
- **주기적(`@Scheduled`)으로 만료된 키 정리** → 메모리 누수 방지
- **서버 1대 전제**이므로 Redis 등 외부 스토어 불필요
- PoW(Altcha)의 인메모리 관리와 **동일한 패턴**으로 일관성 유지

> 데이터 구조 예시
>
> | Key | Value |
> |---|---|
> | 클라이언트 IP | 시도 횟수, 윈도우 시작시각, 차단 단계, 차단 해제시각 |

---

## 5. 서버 설정 (application.yml)

```yaml
security:
  rate-limit:
    window-minutes: 10        # 카운팅 윈도우 (10분)
    max-attempts: 10          # 윈도우 내 허용 횟수
    block-stage-1-minutes: 5  # 1단계 차단
    block-stage-2-minutes: 15 # 2단계 차단
    block-stage-max-minutes: 30 # 3단계+ 차단 상한
    cleanup-interval-seconds: 60 # 만료 키 정리 주기
```

---

## 6. 응답 사양 (429)

- 차단 중 요청에는 **HTTP 429 (Too Many Requests)** 반환
- **`Retry-After` 헤더**에 남은 차단 시간(초)을 포함 → 프론트엔드가 카운트다운 UI로 활용(무의미한 재시도 억제)
- 응답 본문은 **일반화된 메시지**만 노출 (차단 단계·내부 정책 상세 노출 금지)

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 300
Content-Type: application/json

{ "message": "요청이 많아 잠시 후 다시 시도해 주세요." }
```

## 7. 다이어그램에 없는 추가 권장 사항

> 아래는 현재 다이어그램에 명시되지 않았지만 실무상 검토가 필요한 항목.

- **식별자 단위 제한 병행**: IP만으로는 IP를 바꿔가며 한 식별자를 노리는 공격을 못 막음. **(식별번호 / 휴대폰) 단위 시도 제한**을 추가 권장
- **내부/화이트리스트 예외**: 헬스체크·내부망·모니터링 IP는 예외 처리.
- **enumeration 방어**: 인증 실패 응답은 어떤 필드가 틀렸는지 노출하지 않는 동일 메시지 + 일정한 응답 시간 유지.
- **로깅·모니터링**: 차단 발생 IP·단계·빈도를 로깅하여 공격 패턴 탐지/임계치 튜닝에 활용.

---

## 8. PoW(Altcha)와의 관계

| 계층 | 방어 수단 | 목적 |
|---|---|---|
| 1차 (공통) | **IP Rate Limit** (본 문서) | 요청 양·속도 제한, 반복 위반 차단 |
| 2차 | **Altcha PoW** | 자동화 공격 비용 상승 |
| 3차 | **3개 정보 일치 + 재사용 방지** | 본인 확인, 토큰 1회용 |

→ API 요청도 **반드시 IP Rate Limit을 먼저 통과한 뒤** PoW·간편인증 흐름을 탄다.
